# Техническое задание

## Разработка веб-сервиса поиска по тексту закона РК

---

### 1. Общие сведения

**Наименование проекта:** Навигатор по нормам права (рабочее название; для конкретной демонстрации — «ЗРБ-Навигатор», по Закону «О реабилитации и банкротстве»).

**Тип проекта:** учебный (демонстрационный проект в рамках семинара по вайб-кодингу).

**Заказчик/пользователь:** сотрудники юридических подразделений, работающие с большим объёмом законодательного текста.

**Формат реализации:** одностраничное веб-приложение (HTML + CSS + JavaScript), без серверной части и без базы данных.

---

### 2. Цель и задачи

**Цель:** сократить время поиска нужной нормы в тексте закона за счёт мгновенной фильтрации статей по ключевому слову или фразе.

**Задачи:**
- обеспечить поиск без необходимости вручную пролистывать документ;
- показать пользователю полный текст всех статей, соответствующих запросу, без дополнительных кликов;
- визуально выделить в тексте совпадающие фрагменты;
- предоставить возможность быстро скопировать текст найденной статьи для использования в служебных документах.

---

### 3. Требования к исходным данным

3.1. Источник — текст Закона РК «О реабилитации и банкротстве» от 7 марта 2014 года № 176-V, предварительно разобранный на структурированный массив статей (номер статьи, название статьи, текст статьи).

3.2. Формат данных — JS-массив объектов (либо аналогичная структура в Markdown/JSON), встраиваемый непосредственно в HTML-файл. Пример структуры одного элемента:

```json
{
  "number": "12",
  "title": "Администратор",
  "text": "1. В качестве администратора могут выступать..."
}
```

3.3. Исключённые статьи (например, 43, 57, 60, 65) включаются в массив данных с текстом об исключении, без пропуска номера — для сохранения целостности нумерации.

3.4. Архитектура данных должна допускать расширение на второй закон (или несколько глав другого закона) без изменения логики поиска — за счёт добавления поля с указанием источника (название закона) для каждой статьи.

---

### 4. Функциональные требования

| № | Требование |
|---|-----------|
| 4.1 | Поле поиска с фильтрацией результатов в реальном времени, без кнопки «Найти» |
| 4.2 | Поиск осуществляется по частичному совпадению — по вхождению введённых символов в любую часть слова (например, запрос «реабилит» находит «реабилитация» и «реабилитационный») |
| 4.3 | Поиск выполняется как по названию статьи, так и по тексту статьи |
| 4.4 | Результаты отображаются сразу с полным текстом каждой найденной статьи (без необходимости клика для раскрытия) |
| 4.5 | Совпадающие фрагменты текста подсвечиваются визуально |
| 4.6 | Счётчик результатов («Найдено: N статей») |
| 4.7 | Кнопка «Скопировать текст статьи» рядом с каждой найденной статьёй (копирует номер и текст статьи в буфер обмена) |
| 4.8 | Заголовок страницы с названием закона и его номером/датой |
| 4.9 | При пустом поисковом запросе — отображение статьи в порядке следования, либо информационное сообщение с предложением начать ввод |
| 4.10 | При отсутствии совпадений — сообщение об отсутствии результатов |

---

### 5. Требования к интерфейсу и дизайну

5.1. Стиль — деловой, минималистичный, без декоративных элементов и анимации, не отвлекающей от содержания.

5.2. Цветовая палитра — на основе тёмно-зелёного цвета:
- основной фон: тёмно-зелёный (ориентир — #1B3A2F);
- акценты и подсветка совпадений: светло-зелёные/мятные тона (ориентир — #4CAF7D, #A8D5BA);
- текст: светлый/белый — для контраста с фоном.

5.3. Шрифт — читаемый, без засечек, деловой (системный sans-serif).

5.4. Интерфейс должен быть понятен пользователю без дополнительной инструкции.

---

### 6. Нефункциональные требования

6.1. Приложение работает полностью локально в браузере — без подключения к интернету (за исключением, при необходимости, загрузки веб-шрифта).

6.2. Отсутствие серверной части, базы данных, необходимости установки или регистрации.

6.3. Результат — единый HTML-файл, открываемый двойным кликом в любом современном браузере.

6.4. Время отклика при вводе в поле поиска — мгновенное (без заметной задержки) при объёме данных до 300 статей.

---

### 7. Ограничения (вне рамок текущей версии)

- отсутствует смысловой (семантический) поиск на основе AI — только совпадение по подстроке;
- отсутствует адаптивная вёрстка под мобильные устройства (при необходимости — отдельное требование);
- отсутствует сохранение истории поисковых запросов между сессиями.

---

### 8. Критерии приёмки

| № | Критерий | Статус |
|---|----------|--------|
| 1 | Поиск находит статьи по частичному вхождению слова | — |
| 2 | Результаты показывают полный текст статьи сразу, без клика | — |
| 3 | Совпадения подсвечены визуально | — |
| 4 | Счётчик результатов корректно отображает количество найденных статей | — |
| 5 | Кнопка копирования переносит текст статьи в буфер обмена | — |
| 6 | Цветовая палитра соответствует тёмно-зелёной схеме | — |
| 7 | Файл открывается и работает без интернет-соединения и установки | — |
